引擎框架：Checkpoint 协议与恢复编排

engine 包的核心编排逻辑 — 所有引擎共享

主循环：engine.Run()

func Run(ctx context.Context, repo StateRepository, eng Engine, taskID string) error {
    // 1. 加载状态（可能是新任务、也可能是断点恢复）
    state, err := loadOrInit(repo, taskID)
    if err != nil { return err }

    // 2. 恢复阶段：如果物理系统超前，先补偿
    //    此时 state 是 Store 中的真相，物理系统必须向它对齐
    if state.Phase == PhaseRunning || state.Phase == PhaseMerging {
        if err := eng.Compensate(ctx, state.CheckpointLSN); err != nil {
            return err  // 补偿失败，调用方决定
        }
    }

    // 3. 主执行循环
    for state.Phase == PhaseRunning || state.Phase == PhaseMerging {
        newLSN, err := eng.Execute(ctx, state)
        if err != nil {
            // 不自动保存状态 — 调用方控制
            return err
        }
        state.CheckpointLSN = newLSN
        state.Progress = eng.Progress(state)

        // 原子持久化 checkpoint
        data, _ := json.Marshal(state)
        if err := repo.Save(ctx, state.TaskID, data); err != nil {
            return fmt.Errorf("checkpoint save failed: %w", err)
        }
    }
    return nil
}

Checkpoint 协议映射

Spec 步骤 框架实现 保证
1. 执行动作 eng.Execute(ctx, state) — 引擎在物理系统产生副作用 引擎内部保证返回的 newLSN ≤ 物理系统实际状态
2. 持久化动作 (可选) 引擎在 Execute 内部自行决定是否 fsync/COMMIT 调用方通过 DataSource/DataTarget 实现控制持久化级别
3. 提交 Checkpoint repo.Save(ctx, taskID, data) — 原子写入 Save 成功 → 新 LSN 持久化；Save 失败 → 返回 error，调用方决定

关键设计决策

  • Compensate 只在恢复路径调用 — 正常执行中物理系统和 Store 同步推进，不需要补偿
  • 框架不重试 — Execute 或 Save 失败直接返回 error，由调用方决定重试/放弃/Resume
  • Save 在每次 Execute 后立即执行 — 这不是批量 checkpoint，每一步都持久化。如果调用方需要批量，可以在 DataSource/DataTarget 层自行缓冲
  • 引擎控制 Phase 转换 — Execute 内部通过修改 state.Phase 表达"我完成了"或"我进入合并阶段了"
